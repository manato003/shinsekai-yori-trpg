## OCR エンジン選定

### manga-ocr は日本語縦書き小説に有効
- `MangaOcr` は縦書き日本語テキストに強い（漫画用だが小説にも機能する）
- Tesseract や汎用 OCR より精度が高い
- GPU (CUDA) 環境で1枚約2.5秒（バッチ推論）。逐次処理では約14秒/枚だった

### Ollama VLM フォールバックは危険（削除済み）
- 同一 GPU 上で manga-ocr と Ollama を暗黙的に同時起動すると VRAM 12GB を奪い合いハングする
- フォールバックを自動起動するのではなく、プレースホルダー出力にして手動確認に委ねるべき
- **教訓**: GPU リソースを共有するモデルの自動フォールバックは明示的 opt-in にすること

## 列検出ロジック

### インク密度によるピーク検出が有効
- `col_ink.sum(axis=0)` でカラムのインク密度を計算
- スムージング（移動平均）後にピーク検出
- 孤立ピーク（章番号・ページ番号）は隣接距離の中央値×1.5 を閾値として除去

### 動的チャンク分割で94.2%達成
- 列高さが 300px を超える場合にチャンク分割
- 単純な等分割ではなく、インク密度最小点（文字間ギャップ）を境界とする
- `BOUNDARY_TOLERANCE = 25px` の範囲でギャップを探索

### 2段組の境界検出
- 画像中央付近（上下 1/6 の範囲）でインク密度が最小の行を境界とする
- ギャップ閾値: `max_val * 0.05`（インク密度が最大値の5%未満）

## 長時間バッチ処理

### チェックポイント/再開パターン
- 処理済みファイル名を `.checkpoint` テキストファイルに1行ずつ追記
- 起動時に checkpoint を読み込み、処理済みをスキップ
- 出力ファイルを append モードで開く（再開時）
- 完了後に checkpoint ファイルを削除
- `out.flush()` を各ページ処理後に呼ぶことでクラッシュ時の損失を最小化

## バッチ推論

### manga-ocr の内部 API を使ったバッチ処理
- `MangaOcr.__call__` は1枚ずつしか処理できないが、内部の `processor`, `model`, `tokenizer` を直接使えばバッチ推論可能
- `torch.stack([tensor1, tensor2, ...]).to(device)` → `model.generate(batch, max_length=300)` で N 枚を1回の推論にまとめる
- 性能: 14秒/枚（逐次） → 2.2-2.5秒/枚（バッチ）、約7倍高速化

### VRAM 安全性: サブバッチ分割パターン
- `MAX_BATCH_SIZE = 64` で上限を設定し、超過時はサブバッチに分割
- サブバッチ後に `del batch, out` で即時 VRAM 解放
- 通常ページ（40-60チャンク）は1回で処理されるため性能低下なし
